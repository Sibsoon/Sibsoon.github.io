Images as Textures (5 pt.)



Modified Files:
  • src/imagetexture.cpp

For the i/o of the images we used an external header library from nothings/stb. This library allows us to read the images in different formats. For our particular task we used stbi_loadf().
For the evaluation, we used the JPEG file shown in the Test Texture image. This image provides a variety of colors and will show differences in scaling and rotation.
The TextureMap is part of the nori Texture interface. The texture map takes two optional arguments, inspired by the checkerboard implementation, scale and delta. These optional parameter allows the user to shift and scale the texture if necessary.
Because we used the same base structure as the checkerboard texture, we only had to add a new texture subclass, the rest will be handled by the existing code.

Test Texture

The first comparison we made is a simple scene consisting of a cornell box and a plane. The plane is textured with the previously introduced texture. In the following figure we compare the texutre rendering from our implementation with the rendering of the mitsuba2 render engine.
It shows that the texture is mapped identical.

Cornell Box with Textured Plane

Next we compare the texutred analytic sphere. The mitsuba2 engine has a slightly different coordinate setup as our implementation. This is most likely due to the coordinate system for the image pixels (i/o functions) and possibly due to the internal uv mapping, since there are different common coordinate system usages in this field.
To match the appearence of the mitsuba2 sphere we applied a delta shift and a scaling to counterweight the difference.
Sometimes we experienced a slight difference of the two images with the exact compensation factors. This is most likely an issue with the discrete pixel calculations made during the process of shifting, scaling and uv mapping.
